## Accuracy bugs (root causes of wrong/random scores)

**Bug 1 — `graders/grader.py` `_compile_results`: uninitialized `time_to_detection`**

If `task_name` is not in the hard-coded tuple AND `raw_times` is empty, the variable is never assigned. Crashes rather than scoring wrong, but it's a landmine.

```python
# BEFORE (line ~113):
if raw_times:
    time_to_detection = float(np.mean(raw_times))
elif task_name in ("task_misinfo", "task_spam", "task_cib"):
    time_to_detection = float(np.mean(counts["lengths"])) if counts["lengths"] else 20.0
# no else → UnboundLocalError on unknown task names

# FIX:
if raw_times:
    time_to_detection = float(np.mean(raw_times))
else:
    time_to_detection = min(
        float(np.mean(counts["lengths"])) if counts["lengths"] else float(max_hops),
        float(max_hops)   # cap so speed_score never goes negative
    )
```

---

**Bug 2 — `graders/grader.py` `normalized_score` task_misinfo: speed_score goes negative → score always 0**

When the baseline agent never fires a remove (or fires late), `time_to_detection = mean_episode_length` which is often ~50–100 steps, far above `max_hops=20`. `1 - 50/20 = -1.5`, clamped to 0, making the speed component always 0.

```python
# BEFORE:
speed_score = 1.0 - (time_to_detection / max_hops)
return float(np.clip(0.6 * f1 + 0.4 * max(0.0, speed_score), 0.0, 1.0))

# FIX:
speed_score = max(0.0, 1.0 - (min(time_to_detection, float(max_hops)) / max_hops))
return float(np.clip(0.6 * f1 + 0.4 * speed_score, 0.0, 1.0))
```

**Bug 3 — `server.py` `grade_task`: wrong fallback dict passed to `normalized_score`**

```python
# BEFORE (line ~161):
task_metrics = results.get("tasks", {}).get(task_name, results)
# If task_name key missing, falls back to the FULL results dict (no f1, recall, etc.)

task_metrics = results.get("tasks", {}).get(task_name, {})

```

**Bug 4 — `sim/content_gen.py` `tick()`: empty frontier never terminates spread**

When `_next_frontier` is empty (no new nodes reachable), `_current_frontier` is NOT updated, so `is_spread_done()` never sees an empty frontier. The engine keeps ticking until `_max_timesteps`, wasting steps and making episode lengths unpredictable.

```python
# BEFORE:
if self._next_frontier:
    self._current_frontier = self._next_frontier
    self._post.hop_count += 1
    self._post.total_reach += len(self._next_frontier)
self._timestep += 1

# FIX:
self._current_frontier = self._next_frontier   # always update, even if empty
if self._next_frontier:
    self._post.hop_count += 1
    self._post.total_reach += len(self._next_frontier)
self._timestep += 1
```

**Bug 5 — `baseline.py` `_infer_task`: all-zeros obs returns `"task_cib"` but is scored as spam**

Terminal observations (all zeros after done) misroute to CIB, but the grader still accumulates wrong metrics.

```python
# BEFORE:
if obs.shape[0] >= OBS_DIM and not np.any(np.abs(obs) > 1e-8):
    return "task_cib"

# FIX — return early with allow instead of a misrouted score:
if not np.any(np.abs(obs) > 1e-8):
    return "task_spam"   # safe default; grader filters terminal steps via done flag
```

**Bug 6 — `graders/grader.py` `evaluate`: entity_id type mismatch silently breaks TP counting**

For task_misinfo, `entity_id = "misinfo_content"` (string). For task_cib, `entity_id` is an int like `42`. Both go through `str(entity_id)` so they become `"misinfo_content"` and `"42"`. But the grader only appends to `detection_times` on the first TP (`ep_tp_added` guard) — if two steps in one episode both see `gt==1` and action==REMOVE, the second is silently dropped, understating TP count.

```python
# FIX — reset ep_tp_added per-entity, not per-episode, for task_cib:
# In the evaluate loop, after the while loop ends, use len(bots_removed) directly
# (already done correctly at episode end) — the bug is ep_tp_added blocking
# detection_times from being appended more than once per episode for task_cib.

# Replace the ep_tp_added guard with:
if act == ACTION_REMOVE and entity_id not in bots_removed:
    bots_removed.add(entity_id)
    if task_name == "task_misinfo":
        metrics[task_name]["detection_times"].append(
            int(step_info.get("hop_count", step_info["episode_step"]))
        )
    else:
        metrics[task_name]["detection_times"].append(step_info["episode_step"])
```

**Bug 7 — `sim/social_graph.py` `get_graph_features`: community_assignment can exceed 1.0**

When only 1 community exists, `max(1 - 1, 1) = 1`, so node at community index 1 gets `1/1 = 1.0`. The clip saves it but any node at index >= 1 maps to 1.0, collapsing all variation.

```python
# FIX:
n_comms = len(self._communities_cache)
if n_comms <= 1:
    community_idx = 0.0
else:
    community_idx = comm_i / (n_comms - 1)
```

After `remove_content()`, `is_spread_done()` is True, so `get_current_hop()` returns `_max_hops` (20). This inflates hop counts in the grader's `detection_times`, making early removals look slow.

```python
if self._content_engine.is_spread_done():

# In TaskMisinfo.step(), before calling content_engine methods:
self._final_hop = self._content_engine.get_current_hop()

# In get_current_hop():
if self._acted:
    return self._final_hop
if self._content_engine is None or self._content_engine.is_spread_done():
    return self._content_engine.get_current_hop() if self._content_engine else 0
return self._content_engine.get_current_hop()
```

**Bug 10 — `env/rewards.py` `_compute_speed_bonus`: ACTION_REDUCE_REACH on a bot gives speed bonus but `_compute_correctness` only gives 0.5**

These are logically inconsistent — speed bonus fires on reduce_reach for bots (implying decisive action) but correctness only gives partial credit. For task_misinfo, reduce_reach IS the decisive action, so it should get full correctness credit.

```python
# FIX in _compute_correctness:
if is_bot:
    if action == ACTION_REMOVE:
        return 1.0
    if action == ACTION_REDUCE_REACH:
        return 0.8   # decisive for misinfo, slightly less than remove
    if action in (ACTION_WARN,):
        return 0.5
    if action == ACTION_ESCALATE:
        return 0.3
    return 0.0
```

**Bug 11 — `server.py` `grade_task`: `collateral_threshold` not passed from task config to grader metrics**

`normalized_score` for task_cib divides by `collateral_threshold` defaulting to 10, but the actual config value (from `configs/task3.yaml`) is also 10 — however inference.yaml sets it to 10 too but the env used in grading uses `task3.yaml` with `collateral_damage_threshold: 10`. The deeper issue is that `task_metrics` returned by `_compile_results` computes `collateral_threshold` as the mean of episode-level thresholds only when `step_info` includes it. If the key is missing, it defaults to 10.0 — which may or may not match the actual configured value.

```python
# FIX in server.py grade endpoint — pass actual task config value:
actual_threshold = float(env._task_cfg.get("collateral_damage_threshold", 10))
if "task_cib" in task_name:
    task_metrics["collateral_threshold"] = actual_threshold
score = grader.normalized_score(task_name, task_metrics)
```

**Enhancement 3 — `/grade/all` endpoint**

Run all three tasks in one call and return a combined score.

```python
@app.get("/grade/all", tags=["grading"])
def grade_all(n_episodes: int = 10, seed: int = 42):
    results = {}
    total = 0.0
    for task in TASK_CONFIG_MAP:
        r = grade_task(task, n_episodes=n_episodes, seed=seed)
        results[task] = r
        total += r["score"]
    results["combined_score"] = round(total / len(TASK_CONFIG_MAP), 4)
    return results
```
