Critical
Task 3 observation layout breaks silently with inference.yaml
What the user sees: Task 3 (CIB) always scores near zero and the baseline agent always picks ACTION_ALLOW during inference runs.

When it happens: Any run using configs/inference.yaml, which sets embedding_dim: 32. TaskCIB builds its observation as [emb_32, graph_feats_4] = 36 dims, padded to 68. Graph features land at indices 32–35 instead of the expected 64–67 (IDX_DEGREE_CENTRALITY, etc.). Every consumer reading indices 64–67 gets zeros.

tasks/task_cib.py + configs/inference.yaml
Fix: in inference.yaml set embedding_dim: 64, or in _load_current_node() pad the embedding to TASK3_EMBEDDING_DIM before concatenating graph features so they always land at indices 64–67.
Critical
Global registry lock freezes all API endpoints during Task 3 initialization
What the user sees: Every request to /reset, /step, or /grade for any task hangs with no response — even Task 1 and Task 2 — while one environment is initializing.

When it happens: The first request to any Task 3 endpoint. get_env_and_lock() holds the global _registry_lock for the entire duration of SocialGuardEnv(...) construction, which can take 2–5 minutes when node2vec embeddings are computed.

server.py · get_env_and_lock()
Fix: check if task_name not in _envs under the lock, release the lock before constructing the env, then re-acquire to store it (double-checked locking pattern).
Major
Invalid API action returns HTTP 500 instead of 400
What the user sees: Submitting an out-of-range action (e.g., action: 99) to POST /step returns a confusing "Internal Server Error" with no explanation.

When it happens: env.step() raises ValueError for invalid actions, but the server only catches RuntimeError (→ 400) and generic Exception (→ 500). ValueError falls into the 500 branch.

server.py · step_env() line ~110
Fix: add except ValueError as exc: raise HTTPException(status_code=400, detail=str(exc)) before the generic except Exception block.
Major
Spectral embedding fallback directly leaks the is_bot label into the observation
What the user sees: When spectral embedding fails (e.g., scipy not installed or graph too small), Task 3 becomes trivially solvable — any agent achieves near-perfect F1 because the fallback embedding encodes 1.0 for bots and 0.0 for real users as the 4th feature dimension.

When it happens: Any time eigsh throws an exception during Task 3 reset. The fallback at line base[i, 3] = 1.0 if bool(attrs.get("is_bot", False)) else 0.0 is literal ground-truth leakage.

tasks/task_cib.py · _compute_embeddings_spectral() fallback block
Fix: replace the 4th fallback feature with a neutral structural signal (e.g., account age ratio) instead of the raw is_bot flag.
Major
Baseline agent scores zero for all CIB nodes when using inference config
What the user sees: The /grade/task_cib endpoint reports near-zero baseline score; the grader comparison shows the RL agent beating a baseline that always allows every node.

When it happens: When inference.yaml is used (embedding_dim=32). _score_task_cib() reads from IDX_DEGREE_CENTRALITY=64, IDX_CLUSTERING_COEFF=65, and IDX_PPH_NORMALIZED=67, but those indices hold zeros — the actual graph features are at 32–35. Score is always 0.0, so the baseline always returns ACTION_ALLOW.

baseline.py · _score_task_cib() + root cause in tasks/task_cib.py
Fix: same as Bug 1 — ensure graph features are always placed at fixed indices 64–67 regardless of embedding_dim.
Major
Multi-episode training loop breaks after first episode completes
What the user sees: When running the environment for multiple episodes (e.g., SB3 training), calling env.step() after a completed episode and before env.reset() returns an all-zeros observation with already_done=True — but this post-terminal step is missing the per-task collateral_count (hardcoded as 0), which corrupts TensorBoard callback metrics for Task 3.

When it happens: SB3's VecEnv wrapper calls one extra step after terminal before resetting; the _is_done fast-path in env.step() returns collateral_count=0 even if the true value is nonzero.

env/env.py · step() _is_done fast-path ~line 165
Fix: read collateral_count from self._task.get_collateral_count() in the fast-path return block rather than hardcoding 0.
Major
greedy_modularity_communities blocks the step loop with O(n² log n) recomputation
What the user sees: Task 3 episodes freeze for several seconds every ~20 node removals — the dashboard stalls and the training loop shows sudden latency spikes.

When it happens: get_graph_features() is called on every step of Task 3. Every 20 removals, _ensure_communities_cache() runs greedy_modularity_communities on the remaining ~400-node graph (synchronously, blocking the step).

sim/social_graph.py · _ensure_communities_cache()
Fix: run community detection in a background thread and cache the last valid result; or replace with a cheaper per-node metric (e.g., connected component label via union-find).
Major
Task 1 grader reports time_to_detection = 0.0 when no bots are ever caught
What the user sees: A poorly-performing model that misses all bots shows time_to_detection: 0.0 in the evaluation JSON — the same as a perfect detector — making the metric misleading in results comparisons.

When it happens: When detection_times is empty for task_spam (no TP in any episode), _compile_results() returns 0.0 via the else: time_to_detection = 0.0 branch instead of a large value indicating no detection.

graders/grader.py · _compile_results() ~line 115
Fix: for task_spam with empty detection_times, set time_to_detection = float(np.mean(counts["lengths"])) (same treatment as task_misinfo) to signal slow/no detection.
