## 🐛 Critical & Major Bugs Found

### CRITICAL

**1. `inference.py` — Wrong env var name in README vs code**
- README says `OPENAI_API_KEY` but code uses `HF_TOKEN`. The hackathon spec says `HF_TOKEN`. Consistent in code but the README description says "OPENAI_API_KEY" in one place — misleading for validators.

**2. `server.py` — Race condition in per-task env registry**
```python
# get_env_and_lock creates lock then RELEASES registry lock before acquiring env lock
# Another thread can sneak in and double-init the same env
with _registry_lock:
    if task_name not in _locks:
        _locks[task_name] = threading.Lock()
# GAP HERE — no env init yet
with _locks[task_name]:
    if task_name not in _envs:  # can be entered twice concurrently
```

**3. `env/env.py` — `step()` after done raises but `_is_done` only set after first step**
- `_is_done` is initialized in `reset()` but the attribute doesn't exist until `reset()` is called. The guard `getattr(self, "_is_done", False)` works, but if someone calls `step()` before `reset()`, the `RuntimeError("reset() must be called before step()")` fires correctly — however `_task is None` check happens before `_is_done` check, so it's fine. **Minor ordering issue but not a crash.**

**4. `tasks/task_cib.py` — `_NODE2VEC_CACHE` is a module-level global dict that grows unbounded in multi-process training**
```python
# Cache bounded to 2 entries but uses module-level dict
# In SubprocVecEnv, each subprocess has its OWN copy → cache never shared → always recomputes
# Cache does nothing for multi-env training
if len(_NODE2VEC_CACHE) > 2:
    _NODE2VEC_CACHE.pop(next(iter(_NODE2VEC_CACHE)))
```

**5. `tasks/task_cib.py` — Embeddings computed with stale graph after `remove_node()`**
- `_embeddings_stale = True` is set but **never acted upon** — `_load_current_node()` always uses `self._embeddings[node_id]` without checking `_embeddings_stale`. The flag is set but never triggers a recompute. This means after removals, embedding quality degrades silently.

**6. `graders/grader.py` — FN counting is wrong for Task 1**
```python
# In the step loop:
elif gt == 1 and act != ACTION_REMOVE:
    metrics[task_name]["fn"] += 1  # fires for EVERY non-remove step on a bot
# A bot that gets ACTION_WARN gets 1 FN per step even if eventually removed
```
This inflates FN counts massively, making recall/F1 artificially low.

**7. `server.py — /grade` endpoint holds the env lock for entire N-episode evaluation**
```python
with lock:
    grader = Grader(env, n_episodes=n_episodes)
    results = grader.evaluate(agent, agent_name="baseline")
```
This blocks `/reset` and `/step` for the entire duration of grading (could be minutes). No timeout. Other requests will hang.

**8. `inference.py` — `signal.SIGALRM` not available on Windows; fallback works but has a subtle bug**
```python
use_signal = hasattr(signal, "SIGALRM")
# ...
if use_signal:
    signal.alarm(0)  # Only cancelled if use_signal=True
# But on Windows, the manual timeout check uses monotonic clock correctly
# The issue: on timeout, emit_end() is called but env.close() may not be called
# because the finally block checks `if env is not None` — but env IS not None after init
# This is actually fine... BUT:
# emit_end() is called INSIDE the TimeoutError handler AND in the finally block
# → [END] line emitted TWICE on timeout
```

**9. `tasks/task_spam.py` — `_load_current()` called with stale index during `is_done()` check**
```python
def step(self, action):
    self._current_idx += 1
    if not self.is_done():
        self._load_current()
# is_done() checks self._current_idx >= len(self._queue)
# But get_observation() after done returns stale previous obs — zero-padding not guaranteed
```

**10. `evaluate.py` — Path traversal vulnerability**
```python
trusted_root = (Path.cwd() / "models").resolve()
# If CWD changes between calls, trusted_root changes
# Symlinks can bypass this check entirely
```

### MAJOR

**11. `dashboard/app.py` — `get_env()` caches env in `st.session_state` but Streamlit reruns clear callbacks**
- When the user changes config in the sidebar, `get_env()` compares `sg_env_config_path` but if Streamlit restarts the Python process (file watcher), the old env is lost without `close()` being called → resource leak.

**12. `sim/social_graph.py` — `_ensure_communities_cache()` recomputes on EVERY step after any `remove_node()`**
- `self._communities_dirty = True` is set on every `remove_node()`. For Task 3 with 500 nodes, `greedy_modularity_communities` is O(n log n) and is called during `get_graph_features()` which is called every step. With 500 steps and removals, this causes significant slowdown.

**13. `tasks/task_cib.py` — Node skip loop can infinite-loop if entire graph is removed**
```python
while (
    self._current_node_idx < len(self._node_order)
    and self._node_order[self._current_node_idx] not in self._graph.graph.nodes()
):
    self._current_node_idx += 1
```
`is_done()` is not checked inside this loop. If the graph is emptied but `is_done()` returns False for some edge case, this loop terminates correctly — but it's fragile.

**14. `graders/grader.py` — `normalized_score()` for `task_misinfo` uses wrong key**
```python
score_misinfo = grader.normalized_score("task_misinfo", {"f1": 0.5, "mean_episode_length": 5, "max_hops": 20})
# But normalized_score() reads `time_to_detection`, not `mean_episode_length`
# The test in test_grader.py passes wrong keys — score silently defaults to 0
```

**15. `env/env.py` — `_count_collateral()` is a fallback that double-counts**
```python
# step() tries task.get_collateral_count() first, then falls back to _count_collateral()
# But _count_collateral() iterates decision_history which includes ALL episodes
# (history is reset in reset() so this is fine per episode, but the 10k cap means
#  old decisions from previous episodes could bleed in if > 10k steps)
```

**16. `tasks/task_misinfo.py` — `legitimacy_score` never updates across ticks**
- Legitimacy is set once from `spread_rate` at reset and never changes. As content spreads and `engagement_ratio` grows, the legitimacy should arguably update — but this is a design gap, not strictly a bug.

**17. `sim/content_gen.py` — `tick()` is called in `TaskMisinfo.step()` AFTER checking `is_spread_done()` but the obs returned still reflects the pre-tick state**
```python
# In TaskMisinfo.step():
if not self._content_engine.is_spread_done():
    self._content_engine.tick()
self._increment_step()
# The NEXT call to get_observation() returns post-tick state ✓
# BUT: get_current_hop() called by env.py BEFORE step() reflects pre-tick hop
# → speed bonus computed with stale hop count
```

**18. `training/train_ppo.py` and `train_dqn.py` — `make_env` is a closure that captures `merged_config_path` by reference**
```python
def make_env(rank: int) -> callable:
    def _init() -> SocialGuardEnv:
        return SocialGuardEnv(merged_config_path, seed_offset=rank)
    return _init
# This is fine as-is, but if merged_config_path is deleted before SubprocVecEnv
# workers start (unlikely), workers will fail with FileNotFoundError silently
```

**19. `server.py` — `/grade` uses `BaselineAgent`, not the submitted agent**
```python
agent = BaselineAgent()
results = grader.evaluate(agent, agent_name="baseline")
```
The `/grade` endpoint always evaluates the **baseline**, not any trained model. This is by design per README, but it means the hackathon validators will only ever see baseline scores via `/grade`. If an agent file is expected, there's no way to pass it via HTTP.

**20. `dashboard/app.py` — `load_model` import inside click handler but `evaluate.py` has path security check that rejects paths outside `models/`**
```python
# evaluate.py:
trusted_root = (Path.cwd() / "models").resolve()
# If the dashboard is opened and a model outside models/ is entered,
# it raises ValueError with no user-friendly error in the dashboard
```

**21. `configs/inference.yaml` — Missing `graph` section keys needed by `SocialGraph`**
```yaml
graph:
  num_nodes: 100
  bot_cluster_size: 20
  embedding_dim: 32
  embedding_method: "spectral"
# Missing: intra_cluster_density, inter_cluster_density
# SocialGraph.__init__ will crash with KeyError
```

**22. `env/env.py` — `apply_overrides()` does NOT reinitialize `_reward_engine`**
```python
self._reward_cfg = self._cfg.get("reward", self._reward_cfg)
# But self._reward_engine is NOT recreated with new reward config
# Curriculum overrides to reward coefficients are silently ignored
```

**23. `tasks/task_cib.py` — `_compute_embeddings_spectral()` silent failure returns random projections**
```python
except Exception as exc:
    logger.warning('Spectral embedding failed (%s); using random projections.', exc)
    rng = np.random.RandomState(42)
    vecs = rng.randn(n, k).astype('float32')
```
Random projections are always the same (seed=42) regardless of the graph — all episodes with fallback get **identical embeddings**, making the task unsolvable.

**24. `graders/grader.py` — `Grader.evaluate()` does not call `env.reset()` at start**
- The first episode calls `env.reset(seed=BASE_SEED + 0)` inside the loop, which is correct. But if the env was left in a terminated state from a previous call, the `info` from the first reset may have stale task info. This is fine because `reset()` reinitializes everything — **not a bug, just noting it's fine.**

**25. `inference.py` — `emit_end()` called in `TimeoutError` handler AND `finally` block → duplicate `[END]` lines**
```python
except TimeoutError:
    last_error = "timeout"
    success = False
    emit_step(step_count + 1, "allow", 0.0, True, "timeout")
# No emit_end here — actually OK
# emit_end() is at the END of the finally block — this is correct
# But wait: the finally block runs after BOTH except handlers
# → emit_end() is always called exactly once. Actually fine.
# Re-checking... yes, this is OK. False alarm on #8 above re: double emit.
```

## Summary of Severity

| Severity | Count | Key Issues |
|---|---|---|
| 🔴 Critical | 10 | FN double-counting (#6), stale embeddings (#5), /grade locks server (#7), missing inference.yaml keys (#21) |
| 🟠 Major | 13 | Communities recompute every step (#12), reward engine not updated on override (#22), spectral fallback fixed seed (#23) |

The **most impactful bugs for hackathon scoring** are:
1. **#21** — `inference.yaml` missing keys → Task 3 inference crashes
2. **#6** — Grader F1 is artificially deflated → scores below baseline
3. **#7** — `/grade` locks the server → validator timeout
4. **#23** — Spectral fallback = random identical embeddings → Task 3 broken
5. **#5** — Stale embeddings flag never acted upon
